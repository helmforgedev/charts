# ArchiveBox Architecture

## Overview

ArchiveBox is a web archiving application that captures URLs into multiple
formats, including HTML, PDF, screenshots, WARC data, and media. In this chart,
the application runs as a single Kubernetes Deployment backed by one persistent
data volume.

The chart favors a simple and explicit architecture:

- One ArchiveBox pod.
- One ClusterIP service.
- One data PVC by default.
- Optional ingress for HTTP access.
- Optional CronJob for S3-compatible backups.

## Runtime Components

### Deployment

The Deployment runs the official ArchiveBox image and starts the web server with
quick initialization enabled. The pod exposes the configured HTTP port and sets
ArchiveBox behavior through environment variables.

The deployment strategy is `Recreate`. This prevents two ArchiveBox pods from
using the same SQLite database and snapshot directory during rollouts.

### Data Volume

The `data` volume is mounted at `/data`. With persistence enabled, it uses a
PersistentVolumeClaim. With persistence disabled, it falls back to `emptyDir`
for ephemeral testing.

The volume contains all important application state:

- SQLite database.
- Archive metadata.
- Captured files and indexes.
- Snapshot artifacts generated by Chromium and other extractors.

### Chromium Shared Memory

ArchiveBox uses browser tooling for full page capture. The chart mounts
`/dev/shm` as a memory-backed `emptyDir` with a `1Gi` size limit to provide
shared memory for Chromium.

### Service and Ingress

The Service exposes the pod internally using the configured `service.port`.
Ingress is optional. When enabled, hosts and TLS are provided through
`ingress.hosts` and `ingress.tls`.

The default ingress class value is `traefik`, matching the HelmForge local test
cluster and chart documentation.

## Configuration Flow

ArchiveBox settings are mapped from Helm values into environment variables:

| Value | Environment variable | Purpose |
| --- | --- | --- |
| `archivebox.allowedHosts` | `ALLOWED_HOSTS` | Accepted hostnames |
| `archivebox.publicIndex` | `PUBLIC_INDEX` | Public index visibility |
| `archivebox.publicSnapshots` | `PUBLIC_SNAPSHOTS` | Snapshot visibility |
| `archivebox.publicAddLinks` | `PUBLIC_ADD_VIEW` | Public URL submission view |
| `archivebox.searchBackendEngine` | `SEARCH_BACKEND_ENGINE` | Search backend |
| `archivebox.mediaMaxSize` | `MEDIA_MAX_SIZE` | Maximum media download size |
| `archivebox.timeout` | `TIMEOUT` | Per-URL archive timeout |
| `archivebox.timezone` | `TZ` | Runtime timezone |

Admin credentials are loaded from a Kubernetes Secret. The chart can create that
Secret or reference an existing Secret.

## Backup Flow

When backups are enabled, the CronJob executes in two phases:

1. The init container creates a compressed tar archive from `/data`.
2. The upload container sends that archive to the configured S3-compatible
   bucket and prefix.

Both containers share an `emptyDir` work directory. The backup job mounts the
ArchiveBox PVC read-only to avoid modifying application data.

## Scaling Boundaries

This chart does not support horizontal application scaling. SQLite and the
ArchiveBox data directory require single-writer access. To increase capacity,
scale vertically by increasing CPU, memory, storage performance, and storage
size.

## Failure Domains

Primary failure domains:

- PVC exhaustion from large snapshot volume.
- Chromium memory pressure during complex captures.
- Ingress host mismatch with `archivebox.allowedHosts`.
- Backup jobs failing because credentials, endpoint, or bucket policy are
  incorrect.

Use Kubernetes events, pod logs, PVC metrics, and CronJob job history as the
first troubleshooting points.
